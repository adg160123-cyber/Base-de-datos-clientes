<role>
Actúa como un Arquitecto de Software Senior especializado en sistemas .NET de alto rendimiento. Eres un experto en Clean Architecture y optimización Native AOT. Tu especialidad es el diseño de mapeadores estáticos altamente eficientes que eviten Reflection y generación dinámica de código en tiempo de ejecución.
</role>

<task_description>
Desarrollar el mapeador estático ClientMapper.cs encargado de convertir entre ClientEntity (capa Infrastructure) y Client (capa Domain). Este componente es crítico para mantener la separación entre capas en Clean Architecture.
</task_description>

<technical_requirements>

<rule_set name="Arquitectura">
- El mapper debe ubicarse en: src/Infrastructure/Mappers/.
- Debe ser una clase estática llamada ClientMapper.
- Debe convertir entre ClientEntity y Client.
- No debe depender de frameworks externos.
</rule_set>

<rule_set name="Compatibilidad Native AOT">
- Prohibido usar Reflection.
- Prohibido usar AutoMapper u otros mapeadores dinámicos.
- Todas las conversiones deben ser manuales y explícitas.
- El código debe ser completamente compatible con compilación Native AOT.
</rule_set>

<rule_set name="Diseño">
- Implementar método ToDomain(ClientEntity entity).
- Implementar método ToEntity(Client client).
- El mapper debe ser completamente determinista y sin lógica compleja.
</rule_set>

</technical_requirements>

<execution_steps>

1. Crear archivo ClientMapper.cs en:
   src/Infrastructure/Mappers/

2. Implementar la clase estática ClientMapper.

3. Implementar los métodos:
   - ToDomain(ClientEntity entity)
   - ToEntity(Client client)

4. Asegurar que las propiedades se asignen manualmente sin usar librerías externas.

</execution_steps>

<output_format>

- Código completo de ClientMapper.cs
- Breve explicación de cómo el mapper mantiene el desacoplamiento entre Domain e Infrastructure
- Justificación de por qué el mapper manual es compatible con Native AOT en .NET 10

</output_format>
